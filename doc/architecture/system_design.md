# ISO8583 Message Generator: システムアーキテクチャ設計書

## 1. 背景と目的
本プロジェクトは、`pyiso8583` ライブラリを利用してISO 8583メッセージ（金融取引メッセージ規格）の生成および解析を行うラッパークラス群を開発するものです。

今後のプロジェクト規模拡大（シミュレータ化、REST API化、パーサーライブラリの入れ替えなど）を見据え、以下の原則に基づいて設計を行います。
- **クリーンアーキテクチャの採用**: ビジネスロジックと外部ライブラリ（インフラストラクチャ）の依存を分離する。
- **メタデータ駆動とコード自動生成**: 128個存在するデータエレメントをコードに直接手書きするのではなく、独立したJSON定義（メタデータ仕様の正本）を用いた自動コード生成により、仕様変更に強く型安全なデータモデルを提供する。
- **SOLID原則（依存関係逆転の原則: DIP）**: ラッパーやユースケースは具象クラスではなく、インターフェース（Protocol / ABC）に依存させる。

## 3. データエレメントのモデル設計とコード自動生成

ISO 8583で定義されるデータエレメント（2〜128列）は、仕様によって各フィールドの桁数や属性（LLVAR, LLLVAR, Fixed型、文字列／バイナリ等）が異なります。これを手動で開発すると変更追従やバグが不可避となるため、以下の仕組みを導入します。

### 3.1 仕様の単一情報源化 (JSON)
データエレメントの仕様変更（桁数変更、フィールドの追加）は、Pythonコード（`src/`）を書き換えるのではなく、非コードアセットである `app/data/schemas/iso8583_fields.json` を更新することで対応します。

### 3.2 メタデータ駆動のコード生成
上記JSON定義を読み込み、PydanticのBaseModelなどをベースにしたPythonモデルファイル（`src/iso8583_manager/core/models/generated/iso_models.py`）を出力するジェネレータ（`generate_models.py`）を運用します。
- 生成されたクラスの属性には `Field(max_length=...)` などの入力制限が動的にマッピングされます。
- オブジェクト生成のタイミングで強力な型バリデーションを実行でき、開発時にはIDEの自動補完・静的型解析が有効になります。

## 4. インターフェース依存（DIP）による責務分離

外部ライブラリ（`pyiso8583`）の変換ロジックをカプセル化するラッパークラス等（`infrastructure/` 下に配置）は、自動生成された具体的なモデルクラス（`Iso8583MessageModel`等）を**直接インポート（依存）しない**アプローチをとります。

代わりに、`core/interfaces/` パッケージに定義された抽象インターフェース `IIso8583Model` （ProtocolまたはABC）を実装の前提とします。
これにより、特定のPydanticモデルなどに業務ロジックやラッパーがロックオンされることを防ぎ、アーキテクチャに対して以下のメリットをもたらします。

- **ライブラリ交換の容易さ**: `pyiso8583` の保守が終了し別ライブラリに乗り換える場合でも、インフラストラクチャ層を新しく作成するだけで済みます。
- **安全な疎結合（デメテルの法則等）**: モデルは自己の検証機能のみを備え、ラッパーはインターフェース越しのデータ変換のみに集中します。
- **テスタビリティの向上（TDD）**: ユースケースからメッセージ生成を呼び出す際、インフラ実装の代わりにモックを注入できるため、完全な単体テストが実現可能となります。
